## Версия 1.0 | Независимый Агент-Валидатор

---

## 1. Общая Архитектура

### 1.1. Позиционирование

Система фактчеккинга — это **независимый верификационный слой** Deep Research Skill, работающий как отдельный агент-валидатор. Она не заменяет исследовательский агент, а проверяет его выводы *ex post*, обеспечивая качественный контроль перед финальной выдачей пользователю.

**Ключевой принцип:** *Trust, but verify* — основной агент доверяет своим источникам, агент-валидатор доверяет только перекрёстно подтверждённым фактам.

### 1.2. Компоненты Системы

```
┌─────────────────────────────────────────────────────────────┐
│                    DEEP RESEARCH SKILL                       │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────────┐      ┌──────────────┐      ┌───────────┐ │
│  │   Research   │─────▶│  Fact Check  │─────▶│  Final    │ │
│  │    Agent     │raw   │    Agent     │rated │  Output   │ │
│  │              │findgs│  (Validator) │findgs│ Formatter │ │
│  └──────────────┘      └──────────────┘      └───────────┘ │
│         │                      │                            │
│         └──────────────────────┘                            │
│              Feedback Loop                                  │
└─────────────────────────────────────────────────────────────┘
```

### 1.3. Design Principles (вдохновлены Fintech Discovery Skill)

| Принцип | Реализация |
|---------|-----------|
| **Cost-aware routing** | Проверка дорогих источников (SEC, платные БД) только для high-stakes утверждений |
| **Quality gates** | Каждое утверждение должно пройти минимум 2 стратегии проверки |
| **Fail fast** | Очевидные ошибки (порядок величин, даты) отсеиваются на Шаге 1 |
| **Transparent scoring** | Все confidence scores калибруются и объясняются |

### 1.4. Heartbeat & Checkpoints (вдохновлены Life Planning Coach)

```
CHECKPOINT: fact_extraction_complete     # После Шага 1
CHECKPOINT: source_evaluation_complete   # После Шага 4
CHECKPOINT: categorization_complete      # После Шага 5
CHECKPOINT: final_report_ready           # После Шага 7

RECOVERY: Если валидатор «застревает» на источнике >30 сек — failover на альтернативный
RECOVERY: Если <50% утверждений удалось проверить — эскалация на ручную проверку
```

## 2. Стратегии Проверки Фактов и Цифр

### 2.1. Primary Verification (Проверка по первоисточникам)

**Определение:** Верификация утверждения по оригинальному источнику, где информация была впервые опубликована.

**Применение:**
- Финансовые отчёты компаний → SEC filings (10-K, 10-Q)
- Научные открытия → оригинальные статьи в рецензируемых журналах
- Официальная статистика → первичные данные статистических агентств
- Законодательство → официальные тексты нормативных актов

**Критерии качества первоисточника:**
- [ ] Авторство ясно установлено
- [ ] Дата публикации/обновления указана
- [ ] Методология сбора данных описана
- [ ] Источник доступен для перепроверки

**Confidence modifier:** +0.30 к базовому score

---

### 2.2. Cross-Reference Verification (Перекрёстная проверка)

**Определение:** Сопоставление утверждения с ≥2 независимыми источниками, не связанными между собой.

**Правила независимости источников:**
- Источник A не цитирует источник B (и наоборот)
- Разные типы организаций (не два новостных агентства из одного холдинга)
- Разные юрисдикции для международных фактов

**Уровни подтверждения:**

| Уровень | Источники | Модификатор |
|---------|-----------|-------------|
| Согласованность | 2+ независимых, совпадают | +0.25 |
| Частичное расхождение | 2+, но с различиями в деталях | +0.10 |
| Противоречие | 2+, противоречат друг другу | −0.20, триггер глубокого анализа |
| Нет подтверждения | Ни один независимый источник не подтверждает | +0.00, флаг "Не удалось проверить" |

**Красные флаги при cross-reference:**
- Все источники ведут к одному первоисточнику (вирусная цитата)
- Источники идентичны по формулировкам (копипаст)
- Временной лаг: новые источники просто копируют старые

---

### 2.3. Recency Check (Проверка актуальности)

**Определение:** Оценка того, не устарело ли утверждение с момента публикации источника.

**Матрица устаревания по доменам:**

| Домен | "Свежее" окно | "Предупреждение" окно | "Устарело" |
|-------|---------------|----------------------|------------|
| Финансы (курсы, отчётность) | <24 часов | 24 часа — 90 дней | >90 дней |
| Технологии (версии, фичи) | <30 дней | 30–180 дней | >180 дней |
| Наука (открытия) | <1 года | 1–3 года | >3 года |
| Экономика (макро) | <90 дней | 90 дней — 1 год | >1 года |
| Право (законы) | <30 дней | 30 дней — 6 месяцев | >6 месяцев |
| Общие факты (история, география) | N/A (вечные) | — | — |

**Флаги устаревания:**
- `STALE_DATA`: данные старше окна "предупреждение"
- `DEPRECATED`: данные в окне "устарело"
- `SUPERSEDED`: есть более новая версия/исправление

**Confidence modifier:** −0.15 за `STALE_DATA`, −0.30 за `DEPRECATED`

---

### 2.4. Context Completeness (Проверка полноты контекста)

**Определение:** Оценка того, искажается ли вывод при удалении важного контекста.

**Типы искажения контекста:**

| Тип | Пример | Категория |
|-----|--------|-----------|
| **Cherry-picking** | Указать рост выручки 50%, но не упомянуть, что это после падения 70% | Неполно |
| **False precision** | "Рынок вырастет на 12.34%" — точность не подтверждается данными | Одна из точек зрения |
| **Missing baseline** | "X убил 1000 человек" без сравнения с альтернативами | Неполно |
| **Temporal mismatch** | Сравнение данных за разные периоды без явного указания | Неверно |
| **Scale omission** | "Компания потеряла $1M" для компании с капитализацией $1T | Неполно |

**Проверочные вопросы для каждого утверждения:**
1. Есть ли обратная сторона медали?
2. Какие данные были бы нужны для оспаривания этого утверждения?
3. Сохранится ли вывод, если добавить дополнительную информацию?

**Confidence modifier:** −0.10 при выявлении cherry-picking, −0.20 при critical context missing

---

### 2.5. Bias Detection (Выявление систематических искажений)

**Определение:** Идентификация систематических отклонений в источниках или формулировках.

**Каталог типов bias:**

| Bias | Описание | Индикаторы |
|------|----------|------------|
| **Confirmation bias** | Источник подбирает данные под вывод | Только supporting evidence, нет counter-evidence |
| **Source bias** | Источник имеет vested interest | Финансирование, аффилиация, бизнес-модель |
| **Framing bias** | Одни и те же факты в разных рамках | Эмоциональная лексика, отбор деталей |
| **Survivorship bias** | Учтены только "выжившие" примеры | Нет данных о неудачных случаях |
| **Recency bias** | Чрезмерный вес последних событий | Игнорирование долгосрочных трендов |

**Система scoring bias intensity:**

```
BIAS_SCORE = Σ(indicator_weight × severity)

0.0–0.2: Незначительный bias, допустим
0.2–0.5: Умеренный bias, требует дополнительных источников
0.5–1.0: Высокий bias, утверждение помечается "Одна из точек зрения"
```

---

### 2.6. Quantitative Validation (Проверка чисел)

**Определение:** Математическая и логическая проверка числовых утверждений.

**Подтипы проверки:**

#### 2.6.1. Order-of-Magnitude Check
Проверка порядка величины на разумность.

```
INPUT: "Население Парижа — 500 000 человек"
REFERENCE: Реальное население Парижа ~2.1M (город), ~12M (агломерация)
VERDICT: Ошибка на порядок — фактически неверно (возможно, устаревшие данные 19 века)
```

#### 2.6.2. Ratio & Proportion Check
Проверка внутренней согласованности отношений.

```
INPUT: "Выручка $100M, чистая прибыль $80M, маржа 20%"
CHECK: Если прибыль $80M при выручке $100M, маржа = 80%, не 20%
VERDICT: Противоречие — требуется уточнение, что именно считалось
```

#### 2.6.3. Growth Rate Consistency
Проверка согласованности ростов.

```
INPUT: "Выручка выросла на 50% за год, но число клиентов упало на 30%"
ANALYSIS: Возможно (рост ARPU), но требует объяснения — флаг для проверки
```

#### 2.6.4. Unit Consistency
Проверка единиц измерения.

```
INPUT: "Расстояние — 150 кг" → единица неверная
INPUT: "Температура — 300°C" → проверить, не Kelvin ли имелось в виду
```

**Confidence modifier:** +0.15 при успешной математической валидации, −0.40 при выявлении арифметической ошибки

## 3. Процесс Проверки (7 Шагов)

### Шаг 1: Извлечение Проверяемых Утверждений

**Цель:** Разложить исследовательский вывод на атомарные, проверяемые утверждения.

**Типы извлекаемых элементов:**

| Тип | Пример | Проверяемость |
|-----|--------|---------------|
| **Факт** | "Python создан в 1991 году" | Высокая |
| **Цифра** | "Капитализация Apple — $3.2T" | Высокая |
| **Вывод** | "ИИ заменит 30% рабочих мест к 2030" | Средняя |
| **Предсказание** | "Биткоин достигнет $200K в 2025" | Низкая |
| **Цитата** | "Как сказал X..." | Высокая (проверка аутентичности) |
| **Сравнение** | "A в 2 раза эффективнее B" | Средняя |

**Правила извлечения:**
- Одно утверждение = один проверяемый факт
- Утверждения с "возможно", "вероятно", "некоторые" → пометка "Одна из точек зрения"
- Утверждения без конкретных чисел/дат → требуют дополнительной спецификации

**Output:** Список утверждений с типом и исходным контекстом.

---

### Шаг 2: Классификация Типа Утверждения

**Матрица классификации:**

```
УТВЕРЖДЕНИЕ
    │
    ├── ФАКТУАЛЬНОЕ ─────────────────┐
    │   ├── Historical fact             │ → Primary + Cross-ref
    │   ├── Current state fact          │ → Primary + Recency
    │   └── Statistical claim           │ → Quantitative + Cross-ref
    │                                   │
    ├── ЧИСЛОВОЕ ─────────────────────┤
    │   ├── Absolute value              │ → Quantitative + Primary
    │   ├── Ratio/percentage            │ → Quantitative + Context
    │   ├── Growth/projection           │ → Quantitative + Recency
    │   └── Comparative claim           │ → Quantitative + Cross-ref
    │                                   │
    ├── ВЫВОДНОЕ ─────────────────────┤
    │   ├── Causal inference            │ → Context + Bias detection
    │   ├── Correlation claim           │ → Context + Bias detection
    │   └── Interpretation              │ → Cross-ref + Bias detection
    │                                   │
    └── ПРЕДСКАЗАТЕЛЬНОЕ ────────────┘
        ├── Expert forecast             → Bias + Context ("одна из точек зрения")
        ├── Model projection            → Quantitative + Context
        └── Speculative claim           → Bias ("одна из точек зрения")
```

**Output:** Каждое утверждение с присвоенным типом и набором применимых стратегий проверки.

---

### Шаг 3: Поиск Подтверждающих/Опровергающих Источников

**Алгоритм поиска по приоритетам:**

```
PRIORITY 1: Авторитетный первоисточник
   ↓ Найден? → Да: использовать. Нет: →
   
PRIORITY 2: Независимый авторитетный источник той же категории
   ↓ Найден? → Да: использовать. Нет: →
   
PRIORITY 3: Уважаемое СМИ с репутацией фактчеккинга
   (Reuters, AP, BBC, Bloomberg, Nature News)
   ↓ Найден? → Да: использовать с пометкой. Нет: →
   
PRIORITY 4: Специализированные базы данных
   ↓ Найден? → Да: использовать. Нет: →
   
PRIORITY 5: Прочие источники
   → Использовать только как supporting evidence
   → При отсутствии источников 1–4: флаг "Не удалось проверить"
```

**Минимальный набор для каждого утверждения:**
- Минимум 1 первоисточник ИЛИ 2 независимых вторичных
- Для спорных/политических тем: минимум 3 источника с разными перспективами
- Для числовых утверждений: источник с первичными данными

**Output:** Набор источников с URL/ссылкой, датой, типом.

---

### Шаг 4: Оценка Качества Источников

**Scorecard источника:**

```
SOURCE_QUALITY_SCORE = Σ(вес × оценка) / максимум

Критерии (вес):
├── Авторитетность издания/организации (0.25)
│   ├── Tier 1: Peer-reviewed journal, official gov source (1.0)
│   ├── Tier 2: Reuters, Bloomberg, Nature News (0.85)
│   ├── Tier 3: Уважаемое отраслевое издание (0.70)
│   ├── Tier 4: Новостной сайт без репутации (0.40)
│   └── Tier 5: Блог, форум, соцсети (0.20)
│
├── Первичность данных (0.20)
│   ├── Первоисточник (1.0)
│   ├── Прямая цитата первоисточника (0.80)
│   ├── Аналитика на основе первичных данных (0.60)
│   └── Пересказ без ссылки на источник (0.20)
│
├── Свежесть (0.20)
│   ├── В пределах окна "свежее" (1.0)
│   ├── В окне "предупреждение" (0.70)
│   ├── В окне "устарело" (0.40)
│   └── Дата не указана (0.20)
│
├── Отсутствие bias (0.20)
│   ├── Независимый источник (1.0)
│   ├── Возможен mild bias (0.70)
│   ├── Заметный bias, но факты проверяемы (0.40)
│   └── Пропаганда/PR (0.10)
│
└── Доступность для верификации (0.15)
    ├── Открытый доступ, прямая ссылка (1.0)
    ├── Требует регистрации (0.70)
    ├── Платный доступ, но abstract доступен (0.50)
    └── Нет прямой ссылки, сложно проверить (0.20)
```

**Пороги качества:**

| Score | Интерпретация | Действие |
|-------|---------------|----------|
| 0.85–1.00 | Отличный | Использовать как anchor source |
| 0.70–0.84 | Хороший | Использовать, желательно дополнительный |
| 0.50–0.69 | Средний | Использовать только с corroboration |
| 0.30–0.49 | Слабый | Не использовать в одиночку |
| 0.00–0.29 | Ненадёжный | Исключить, найти альтернативу |

**Output:** Каждый источник с SOURCE_QUALITY_SCORE.

---

### Шаг 5: Присвоение Категории (6-Категориальная Система)

**Алгоритм принятия решения:**

```
НАЧАЛО
 │
 ├── Источники найдены и подтверждают? ──ДА──▶ [ВЕРНО]
 │   └── Полнота контекста OK?
 │       └── НЕТ ──▶ [НЕПОЛНО]
 │
 ├── Источники ПРОТИВОРЕЧАТ? ──ДА──▶ [НЕВЕРНО]
 │
 ├── Источники подтверждали, но данные УСТАРЕЛИ? ──ДА──▶ [УСТАРЕЛО]
 │
 ├── Есть альтернативные позиции, не упомянутые? ──ДА──▶ [ОДНА ИЗ ТОЧЕК ЗРЕНИЯ]
 │   (bias detection сработал ИЛИ нет консенсуса)
 │
 └── Источники НЕ НАЙДЕНЫ? ──ДА──▶ [НЕ УДАЛОСЬ ПРОВЕРИТЬ]
```

**Output:** Каждое утверждение с категорией, confidence score, обоснованием.

## 4. Система Категоризации

### 4.1. Шесть Категорий

---

#### ✅ ВЕРНО — Подтверждается Источниками

| Атрибут | Описание |
|---------|----------|
| **Определение** | Утверждение прямо подтверждается авторитетными первичными или независимыми вторичными источниками без оговорок |
| **Критерии** | ≥1 первоисточник напрямую подтверждает ИЛИ ≥2 независимых Tier-2+ источника согласуются; контекст полный; данные свежие |
| **Визуальное обозначение** | ✅ Зелёный |
| **Частота в реальных исследованиях** | 55–70% |
| **Действие** | Включается в финальный отчёт без изменений |

**Примеры:**
- "Столица Франции — Париж" → ✅ Верно (общепризнанный факт)
- "Apple Inc. капитализация $3.4T на декабрь 2024" → ✅ Верно (подтверждается Yahoo Finance, Bloomberg)

---

#### ❌ НЕВЕРНО — Противоречит Источникам

| Атрибут | Описание |
|---------|----------|
| **Определение** | Утверждение прямо противоречит достоверным источникам, либо содержит внутренние логические ошибки |
| **Критерии** | Авторитетный источник прямо опровергает; ИЛИ математическая несостыковка; ИЛИ дата/событие невозможны |
| **Визуальное обозначение** | ❌ Красный |
| **Частота в реальных исследованиях** | 5–15% |
| **Действие** | **Исключается** из финального отчёта; исследовательский агент уведомляется для пересмотра |

**Примеры:**
- "Первая мировая война закончилась в 1925 году" → ❌ Неверно (закончилась в 1918)
- "Компания X имеет выручку $100M" при SEC filing с $45M → ❌ Неверно

---

#### ⏰ УСТАРЕЛО — Было Верно, Ситуация Изменилась

| Атрибут | Описание |
|---------|----------|
| **Определение** | Утверждение было корректным на момент публикации источника, но с тех пор ситуация изменилась |
| **Критерии** | Источник подтверждает факт для своего времени; но есть более свежие данные, опровергающие актуальность |
| **Визуальное обозначение** | ⏰ Оранжевый |
| **Частота в реальных исследованиях** | 5–12% |
| **Действие** | Помечается с указанием актуального состояния; исследовательский агент обновляет данные |

**Примеры:**
- "CEO компании Y — Джон Смит" (источник 2022) → ⏰ Устарело (с 2024 CEO — Джейн Доу)
- "Ставка ФРС — 5.25%" (источник май 2024) → ⏰ Устарело (в декабре 2024 — 4.50%)

---

#### ⚠️ НЕПОЛНО — Факт Верный, Контекст Искажает Вывод

| Атрибут | Описание |
|---------|----------|
| **Определение** | Фактическая часть утверждения верна, но отсутствие важного контекста ведёт к искажению интерпретации |
| **Критерии** | Основное утверждение подтверждается; но cherry-picking, missing baseline, или scale omission искажает вывод |
| **Визуальное обозначение** | ⚠️ Жёлтый |
| **Частота в реальных исследованиях** | 10–20% |
| **Действие** | Включается с дополнением контекста; исследовательский агент переписывает формулировку |

**Примеры:**
- "Компания Z увеличила выручку на 200%" (но база была $10K, а абсолютный рост — $20K) → ⚠️ Неполно
- "Страна А имеет высокий уровень преступности" (без данных, что он снизился на 50% за 10 лет) → ⚠️ Неполно

---

#### 🔮 ОДНА ИЗ ТОЧЕК ЗРЕНИЯ — Есть Альтернативные Позиции

| Атрибут | Описание |
|---------|----------|
| **Определение** | Утверждение отражает одну из нескольких легитимных интерпретаций; существуют уважаемые альтернативные позиции |
| **Критерии** | Нет консенсуса в авторитетных источниках; ИЛИ bias score >0.5; ИЛИ предсказательное/интерпретационное утверждение без возможности прямой верификации |
| **Визуальное обозначение** | 🔮 Фиолетовый |
| **Частота в реальных исследованиях** | 8–18% |
| **Действие** | Переформулируется как "по мнению...", "согласно одной из интерпретаций..."; альтернативные позиции добавляются |

**Примеры:**
- "Экономика войдёт в рецессию в 2025 году" → 🔮 (есть прогнозы и за, и против)
- "Причиной глобального потепления являются только выбросы CO₂" → 🔮 (есть дискуссия о вкладе других факторов)

---

#### ❓ НЕ УДАЛОСЬ ПРОВЕРИТЬ — Требуется Ручная Проверка

| Атрибут | Описание |
|---------|----------|
| **Определение** | Не удалось найти достаточно авторитетных источников для подтверждения или опровержения |
| **Критерии** | Нет источников Tier 1–3; ИЛИ утверждение слишком специфично; ИЛИ требует доступа к закрытым базам данных |
| **Визуальное обозначение** | ❓ Серый |
| **Частота в реальных исследованиях** | 3–10% |
| **Действие** | Помечается явно; либо исключается, либо включается с флагом "не подтверждено" |

**Примеры:**
- "Компания X планирует приобретение Y в Q1 2025" (инсайдерская информация) → ❓
- "Внутренняя метрика Z для продукта W составляет 99.9%" (нет публичных данных) → ❓

## 6. Агент-Валидатор

### 6.1. Роль и Позиционирование

**Название:** `FactCheck Agent (FCA)`
**Версия:** 1.0
**Статус:** Независимый агент, не подчиняется Research Agent

**Роль в экосистеме:**
- Получает "сырые" findings от Research Agent
- Проводит полную верификацию по описанным выше стратегиям
- Возвращает **размеченный результат** с категориями и обоснованиями
- Имеет **право вето** — может заблокировать включение неверных утверждений

### 6.2. Принципы Работы

#### Принцип 1: Непристрастность (Impartiality)
```
FCA не имеет "предпочтительного" исхода проверки.
Цель — не подтвердить или опровергнуть, а установить факт.
Если Research Agent утверждает X, а источники говорят Y — FCA выбирает Y.
```

#### Принцип 2: Требовательность (Rigor)
```
"Достаточно хорошо" — недостаточно.
Каждое утверждение должно выдержать scrutiny.
Если сомнение остаётся — категория понижается.
```

#### Принцип 3: Прозрачность (Transparency)
```
Каждое решение объясняется.
Пользователь видит: что проверялось, по каким источникам, с каким результатом.
Никаких "чёрных ящиков" в верификации.
```

#### Принцип 4: Экономическая Эффективность (Cost Awareness)
```
High-stakes утверждения → полная проверка (все 6 стратегий)
Medium-stakes → стандартный набор (3–4 стратегии)
Low-stakes → быстрая проверка (2 стратегии)
Никаких дорогих вызовов без обоснования.
```

### 6.4. Recovery & Heartbeat (вдохновлено Life Planning Coach)

```
HEARTBEAT_INTERVAL: 30 секунд
MAX_SOURCE_WAIT_TIME: 30 секунд (per source)
MAX_TOTAL_TIME: 10 минут (per batch)

RECOVERY_SCENARIOS:

1. Источник недоступен (>30 сек)
   → FAILOVER: переключиться на альтернативный источник
   → LOG: записать timeout для последующей аналитики

2. <50% утверждений удалось проверить
   → ESCALATION: пометить батч как "требует ручной проверки"
   → NOTIFY: уведомить пользователя о ограничениях

3. Обнаружено критическое количество ❌ Неверно (>20%)
   → ESCALATION: запросить пересмотр у Research Agent
   → BLOCK: не передавать пользователю до исправления

4. Несогласованность между стратегиями
   → DEEP_DIVE: запустить дополнительные проверки
   → CONFLICT_RESOLUTION: применить tie-breaking rules
```

### 6.5. Tie-Breaking Rules

При конфликте между стратегиями:

```
1. Primary beats secondary (первоисточник > вторичный)
2. Fresh beats stale (свежие данные > устаревшие)
3. Quantitative beats qualitative (цифры > интерпретации)
4. Multiple beats single (множественные источники > одиночный)
5. Transparent beats opaque (прозрачная методология > неизвестная)
```

## 7. Quality Metrics

### 7.1. Основные Метрики

#### 7.1.1. Verification Coverage (Охват Проверки)

```
Verification Coverage = (Число проверенных утверждений / Общее число утверждений) × 100%

Thresholds:
├── 🟢 Отличный: ≥ 90%
├── 🟡 Хороший: 75–89%
├── 🟠 Приемлемый: 60–74%
└── 🔴 Требует внимания: < 60%

Action: Если < 75% — эскалация на ручную проверку.
```

#### 7.1.2. Accuracy Rate (Доля Верных)

```
Accuracy Rate = (Число ✅ утверждений / Число проверенных) × 100%

Важно: Это метрика Research Agent, не FCA.
FCA измеряет, сколько из проверенных оказались верными.

Thresholds:
├── 🟢 Отличный: ≥ 85%
├── 🟡 Хороший: 70–84%
├── 🟠 Приемлемый: 55–69%
└── 🔴 Требует внимания: < 55%

Action: Если < 70% — пересмотр модели/промпта Research Agent.
```

#### 7.1.3. Source Diversity (Разнообразие Источников)

```
Source Diversity Index = Уникальные типы источников / Всего использовано

Типы:
├── Первоисточники (gov, SEC, научные журналы)
├── Новостные агентства (Reuters, AP, Bloomberg)
├── Аналитические (Gartner, McKinsey)
├── Сообщества (GitHub, Stack Overflow)
├── Базы данных (Statista, WB, IMF)
└── Другие

Thresholds:
├── 🟢 Отличный: ≥ 4 типа
├── 🟡 Хороший: 3 типа
├── 🟠 Приемлемый: 2 типа
└── 🔴 Требует внимания: 1 тип

Action: Если < 3 типа — дополнительный поиск по недостающим категориям.
```

#### 7.1.4. Confidence Calibration (Калибровка Уверенности)

```
Идеальная калибровка: Confidence score ≈ реальная вероятность быть верным.

Проверка:
1. Разбить утверждения на бины по confidence (0.9-1.0, 0.8-0.9, ...)
2. Для каждого бина: вычислить % действительно верных
3. Сравнить с серединой бина

Calibration Error = |predicted_confidence - actual_accuracy|

Thresholds:
├── 🟢 Отличный: < 0.05
├── 🟡 Хороший: 0.05–0.10
├── 🟠 Приемлемый: 0.10–0.15
└── 🔴 Требует внимания: > 0.15

Action: Если > 0.10 — перекалибровка scoring system.
```

### 7.2. Метрики Процесса

| Метрика | Описание | Целевое значение |
|---------|----------|-----------------|
| **Avg. verification time** | Среднее время на одно утверждение | < 60 сек |
| **Source retrieval rate** | % успешных попыток найти источник | > 80% |
| **False positive rate** | % ✅, которые на самом деле ❌ | < 3% |
| **False negative rate** | % ❌/⏰/⚠️, которые на самом деле ✅ | < 5% |
| **Escalation rate** | % батчей, ушедших на ручную проверку | < 10% |

### 8.3. Re-check Loop

```
Максимум 3 итерации исправлений:

Итерация 1: Research Agent получает feedback → исправляет
            ↓
            FCA re-check
            ↓
Итерация 2: Если остались ❌/⏰/⚠️ → повторная коррекция
            ↓
            FCA re-check
            ↓
Итерация 3: Если всё ещё есть проблемы → ESCALATION
            ↓
            Ручная проверка + пометка в выводе
```

### 8.4. Cost-Aware Routing (Fintech Skill Pattern)

```
STAKES ОПРЕДЕЛЕНИЕ:

HIGH STAKES (полная проверка, все стратегии):
├── Финансовые цифры, влияющие на инвестиционные решения
├── Медицинские утверждения, влияющие на здоровье
├── Юридические интерпретации с последствиями
└── Любое утверждение с confidence > 0.9 от Research Agent

MEDIUM STAKES (стандартный набор):
├── Общие рыночные тренды
├── Технические характеристики продуктов
├── Экономические обзоры
└── Утверждения с confidence 0.7–0.9

LOW STAKES (быстрая проверка):
├── Общие факты, широко известные
├── Исторические даты
├── Биографические данные
└── Утверждения с confidence < 0.7

COST LIMITS:
├── Max $0.50 per high-stakes claim
├── Max $0.20 per medium-stakes claim
├── Max $0.05 per low-stakes claim
└── Max $10.00 per full research batch
```

### 9.2. Псевдокод Основного Цикла

```python
async def factcheck_pipeline(findings: list, domain: str) -> VerificationReport:
    
    # STAGE 0: Routing
    for finding in findings:
        finding.stakes = assess_stakes(finding, domain)
    
    # STAGE 1: Extraction
    atomic_claims = []
    for finding in findings:
        claims = extract_claims(finding.text)
        atomic_claims.extend(claims)
    checkpoint("fact_extraction_complete")
    
    # STAGE 2: Classification
    for claim in atomic_claims:
        claim.type = classify_claim(claim.text)
        claim.strategies = select_strategies(claim.type, claim.stakes)
    
    # STAGE 3-4: Source search & evaluation
    for claim in atomic_claims:
        for strategy in claim.strategies:
            sources = await search_sources(claim, strategy, domain)
            for source in sources:
                source.quality_score = calculate_source_quality(source)
            claim.sources.extend(sources)
    checkpoint("source_evaluation_complete")
    
    # STAGE 5: Categorization
    for claim in atomic_claims:
        claim.category = categorize(claim)
        claim.confidence = calculate_confidence(claim)
    checkpoint("categorization_complete")
    
    # STAGE 6: Documentation
    for claim in atomic_claims:
        claim.citation = generate_citation(claim)
    
    # STAGE 7: Report
    report = generate_report(atomic_claims)
    checkpoint("final_report_ready")
    
    # STAGE 8: Feedback
    if report.uncertain_rate > 0.5:
        escalate("manual_review_required")
    
    return report
```
