# Release Playbook — DResearch-Skill

Практический чек-лист подготовки релиза. Выжимка уроков v1.6.0 (порт из
OpenResearcher) и v1.7.0 / v1.8.0 (порт из hyperresearch, 9 REQ, 0 регрессий): что делать,
чтобы релиз был чистым и повторяемым, и какие грабли уже известны. Держать в
актуальном состоянии — обновлять после каждого релиза.

---

## 1. Схема ведения фичи «под ключ»

Каждый REQ проходит харнесс-loop (через `Workflow`), модель подбирается по роли:

| Стадия | Модель | Что делает |
|--------|--------|------------|
| Spec | Opus | формализует требование, критерии приёмки |
| Tests-red | Sonnet | пишет тесты ПЕРВЫМИ — они должны упасть |
| Implement | Sonnet | минимальная реализация до зелёных тестов |
| Review | Opus | ревью диффа против критериев |
| Root-cause (условно) | параллельно | при любых ошибках — см. §4 |

Инварианты, которые проверяем на каждой фиче:
- **determinism byte-identical** — вывод не меняется побайтно без причины;
- **обратная совместимость** — новое поле опционально и выпадает из сериализации при `None`;
- **0 регрессий** — золотой корпус и полный прогон зелёные.

## 2. Тесты: smoke-first + таймауты

Не запускать тяжёлые прогоны сразу. Порядок:

1. **Smoke** (быстро, ловит очевидные поломки и зависания).
2. Только потом — полный прогон.

Каждый прогон оборачивать в таймаут, чтобы зависание не стопорило работу:

```powershell
$job = Start-Job { python -m pytest tests/ }
if (Wait-Job $job -Timeout 120) { Receive-Job $job } else { Stop-Job $job; "TIMEOUT" }
```

Калибровка из v1.6.0: SMOKE ~45s, FULL_ENGINE ~120s, FULL_BENCH ~60s. На деле
весь сьют движка идёт ~2–3s, bench ~1s (v1.7.0) — таймаут это дешёвая страховка,
а не узкое место. Гейт детерминизма — прогнать дважды и сравнить хэш:
`bench.trust`×2 + `bench.quality`×2 → `Get-FileHash` diff; плюс
`evals/ci_regression.py` (golden-corpus).

## 3. Сквозная трассировка

`docs/TRACE_*.md` — матрица **REQ → критерий приёмки → тест**. Правило:
**пункт не считается принятым без строки трассировки** и статуса 🟩. Ссылка на
тест обязательна.

## 4. Ошибки → корневая причина, а не костыль

При любых ошибках (упавшие тесты, зависания, регрессии, findings ревью):
запустить **параллельного** агента, который (1) собирает ошибки, (2) ищет
корневые причины, (3) проектирует **системную защиту** (guard/инвариант/тест) —
«как сделать, чтобы это никогда не повторилось». Маппить на корни из
`docs/BUGLOG.md`. Правило проекта: *a bug without a new guard is a bug that can
recur*.

## 5. Ревью перед релизом — в ЧИСТОМ контексте

Финальное ревью проводить в **новом контекстном окне** с минимально необходимым
контекстом (только диффы + критерии приёмки, без истории реализации). Даёт
независимый взгляд и ловит scope-creep. Вердикт **GO / NO-GO**. Findings
закрывать системно.

## 7. Байт-идентичные фичи в determinism-gated движке

Движок под golden-corpus + byte-identical determinism replay. Как добавлять фичи
БЕЗ регрессий (проверено на v1.7.0 — 7 REQ, 0 регрессий):

- **Новое поле модели → `Optional[...] = None`.** `_jsonable` дропает None →
  сериализация байт-идентична, checkpoint back-compat без bump `CHECKPOINT_VERSION`
  (напр. `Claim.citation_spans`, `Source.retracted`).
- **Новый гейт, меняющий отчёт → привязать к opt-in сигналу.** quote-integrity
  срабатывает только на claim'ах со `citation_spans` → legacy/golden не тронуты
  (перед стартом `grep` сигнал в golden — его там быть не должно).
- **Аудит без права на правку → read-only CLI-verb, НЕ вплетать в `render`.**
  numeric/instruction-coverage → отчёт всегда байт-идентичен (тест-guard: `report.py`
  не импортирует модуль).
- **Изменение общей функции → no-op на legacy.** independence-тайбрейкер в
  `resolve_conflict` срабатывает только если ОБЕ стороны заскорены, иначе skip →
  старые данные без изменений.
- **Тяжёлое вычисление → отдельный verb; `run_pipeline` его не вызывает** →
  default-пайплайн и golden неизменны.

## 8. Добавление CLI-verb'а (reachability guard)

Verb «не существует», пока он не проходит guard. Синхронно правь три места, иначе
CI красный:

1. `engine/cli.py` — `_cmd_<verb>` + регистрация субпарсера + запись в `CAPABILITY_VERBS`.
2. `tests/test_phase15_reachability.py` — тот же ключ в кортеже `CURATED_CAPABILITIES`
   (тест на set-equality с `CAPABILITY_VERBS`).
3. `SKILL.md` — задокументировать `engine <verb>` (docs↔CLI guard проверяет, что
   каждый упомянутый verb реально зарегистрирован; JSON-in ключи сверяются для
   snapshot-verb'ов из `_KEY_CHECK_VERBS`).

## 9. Агрегаторы и gate'ы: зеркаль реальный вывод + честные ярлыки

Урок v1.8.0 (H8 ship-gate, H9 реконсиляция) — два способа, которыми аналитическая поверхность начинает ЛГАТЬ:

- **Gate обязан считать ровно то множество, что реально попадает в вывод.**
  Если «shippable»-множество gate'а расходится с тем, что рендерит `report.py`,
  вердикт лжёт: ложный NO-GO (отверг годный отчёт) или ложный GO (пропустил
  дефект). Проверять зеркало ПО КАЖДОМУ измерению: completeness по `shipped`
  (findings ИЛИ corrections, а не только findings); citation-density по
  РЕЗОЛВЯЩИМСЯ источникам (как `report._cites`), а не по truthiness поля
  `c.sources`; warning-проверки — по shipped-подмножеству. Ревью REQ'а с gate'ом
  должно в явном виде перепроверить эквивалентность gate ↔ вывод (обе стороны).
- **Честные ярлыки без catch-all.** Метка не должна утверждать больше, чем есть.
  Единодушное опровержение (all-FALSE) — это `refuted`, НЕ `disputed`; одни
  мнения — `inconclusive`. Согласие ВЕРДИКТОВ не должно прятать расхождение
  ЗНАЧЕНИЙ (флаг `numeric_divergence`). Ветка «иначе → X» — типовой источник
  дезинформации: перечисляй случаи явно, а не сваливай в один ярлык.

## Известные грабли окружения

| Симптом | Причина | Обход |
|---------|---------|-------|
| Edit/Write падают на `Y:` с `fchmod` ENOENT | `Y:` — vmware shared-folder mount | python-патч-скрипт в scratchpad + запуск, либо `Copy-Item -Force` |
| Git «dubious ownership» | тот же mount | `git config --global --add safe.directory '%(prefix)///vmware-host/Shared Folders/GitHub/DResearch-Skill'` (один раз) |
| Коммит «Author identity unknown» | git-идентичность не настроена | задать `user.name`/`user.email`, при необходимости `commit --amend --reset-author` |
| `Bash`-инструмент на `Y:` молча возвращает пусто (особенно `cd` в `Y:`) | mount-специфика | использовать `PowerShell` (основной шелл), не `Bash` |
| subprocess-тест читает вывод движка как `None` / `UnicodeDecodeError` | движок печатает UTF-8, `subprocess(text=True)` декодит локалью (cp1252) | `subprocess.run(..., encoding="utf-8")` |
| patch-скрипт: `anchor not found` при правке многострочного литерала | значение в файле на ОДНОЙ строке (напр. кортеж), а якорь ждал перенос | матчить короткую подстроку без завязки на `\n`; сверять реальный формат перед правкой |
| CRLF→LF предупреждения git | нормализация переводов строк | безвредно, игнорировать |
